### 2.3 为什么不能在手机上实现Nanite或Dreams？

#### 技术时间线对比
```
GPU-Driven Rendering (SIGGRAPH 2015) ────────► 8年前
SDF Ray-tracing - Claybook (GDC 2018) ───────► 5年前
```

#### Nintendo Switch vs 移动手机对比

| 特性 | Nintendo Switch | 移动GPU (底部50%+) |
|------|-----------------|-------------------|
| 峰值算力 | ~200 GFLOP/s | 相近 |
| 内存带宽 | ~20 GB/s | 相近 |
| **通用内存读写** | 快速 | **仅16KB统一缓冲区！SSBO很慢！** |
| **共享内存** | 快速且大 | **小或模拟的** |
| **原子操作/波内指令** | 快速 | **支持率<10%** |
| **寄存器文件/缓存** | 大 | **避免复杂着色器** |
| **64位原子操作** | 支持 | **完全不支持64位整数** |

**核心结论**：50%+的移动手机设计目标是高效运行现有的GLES 3.0游戏，而非现代PC图形技术。

## 三、设计阶段 (Design)

### 3.1 正确的平台抽象层级

#### 传统架构 vs HypeHype架构

```
传统架构 (OLD)                    HypeHype架构 (NEW)
┌─────────────────┐              ┌─────────────────┐
│  业务逻辑        │              │  业务逻辑        │
├─────────────────┤              ├─────────────────┤
│  数据模型        │              │  数据模型        │
├─────────────────┤              ├─────────────────┤
│  高级渲染        │              │  高级渲染        │
├─────────────────┤              ├─────────────────┤
│  着色器         │              │  着色器         │
├─────────────────┤              ├─────────────────┤
│  低级渲染        │              │  低级+中级渲染   │
├─────────────────┤              ├─────────────────┤
│  平台特定GFX API │              │  平台特定GFX API │
└─────────────────┘              └─────────────────┘
```

**传统架构问题**：
- 新特性支持困难
- API膨胀
- 快速路径难以优化
- 维护困难
- 依赖过多
- 平台一致性问题
- 同步发布困难

#### HypeHype解决方案：最小平台抽象

```
┌─────────────────────────────────────────────────────────────┐
│  核心原则：薄的底层图形API包装器                              │
├─────────────────────────────────────────────────────────────┤
│  1. 找到功能的共同集合和差异点                                 │
│     • 交叉参考Vulkan、Metal和WebGPU文档                       │
│     • 设计性能最优的差异抽象方式                               │
│     • Metal 2.0特性：Placement heaps, Argument buffers, Fences│
├─────────────────────────────────────────────────────────────┤
│  2. 剔除已弃用的功能                                          │
│     • Transform feedback                                      │
│     • Triangle strips, fans                                   │
│     • Geometry shaders, HW tessellation                       │
│     • Vertex buffers? (某些移动设备仍受益)                     │
├─────────────────────────────────────────────────────────────┤
│  3. 单一着色器集合                                            │
│     • GLSL + SPIRV-Cross交叉编译                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 低级API设计目标

#### 避免高层概念侵入低级API
| 不做的事情 | 原因 |
|-----------|------|
| 没有网格或材质概念 | 可用VB+IB和绑定组表示 |
| 没有自动数据设置或强制数据布局 | 保持灵活性 |
| 没有固定绘制算法 | 传统、实例化、未来GPU驱动？ |
| 没有磁盘数据加载 | 保持纯粹 |

#### "零"额外API开销
```
设计核心支柱：像DX11一样易用，但像手写优化的DX12一样快

错误方案：在代码库中实现DX11驱动
```

**潜在性能陷阱**：
- 细粒度输入、渲染状态和数据拷贝
- 资源状态跟踪、影子状态
- PSO+渲染状态和绑定组缓存（哈希表）
- 软件命令缓冲区

### 3.3 迭代API设计过程

#### 传统 vs 敏捷+测试驱动

| 传统方法 | 敏捷+TDD |
|---------|---------|
| 大型技术设计文档 | 关注下一个Sprint需要什么 |
| 按计划拆分任务 | 实现经过测试的小型模块化代码 |
| 先设计后编码 | |

**传统方法问题**：计划过早锁定、程序员太晚发现架构问题、重构影响生产

**敏捷方法问题**：只见树木不见森林、好零件≠好架构、难以丢弃100%测试覆盖的生产就绪代码

#### HypeHype解决方案：迭代API设计流程

```
┌─────────────────────────────────────────────────────────────┐
│  迭代流程                                                    │
├─────────────────────────────────────────────────────────────┤
│  1. 编写模拟用户层代码                                        │
│     • 创建资源：纹理、着色器、缓冲区、渲染通道等              │
│     • 用有效数据设置资源                                      │
│     • 使用资源渲染完整帧（+动画）                             │
├─────────────────────────────────────────────────────────────┤
│  2. 编写模拟图形平台API                                       │
│     • 还不写任何后端实现代码                                  │
│     • 与模拟用户层代码一起编译以进行语法检查                  │
│     • 程序还不能链接或运行——这完全没问题！                    │
├─────────────────────────────────────────────────────────────┤
│  3. 迭代直到满意                                              │
│     • 需要时添加模拟用例以提高覆盖率                          │
│     • 出现问题时立即进行大型架构重构                          │
│     • 我们能干净地实现所有图形API吗？最优地抽象差异？         │
│     • 性能好吗？没有分配、拷贝、映射查找等...                 │
├─────────────────────────────────────────────────────────────┤
│  4. 最后：实现平台后端                                        │
│     • 如果发现问题立即重构！                                  │
│     • Vulkan/Metal API验证层 == 大型预存在测试套件            │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 以正确的频率和粒度做事

#### 时间相干性原理
```
• 我们每秒渲染相同的游戏世界60次
• 相机平滑移动（大部分时间）
• ~90%的数据与上一帧相比未改变
```

#### 操作频率分类

| 频率 | 操作示例 |
|------|---------|
| **一次性** | 加载游戏世界（+所有烘焙数据） |
| **低频** | 加载网格、纹理、材质或着色器；更改材质纹理绑定；更改对象网格、材质、着色器变体或渲染状态 |
| **中频** | 更改相机和太阳属性；修改对象/材质颜色（<10%对象）；修改对象变换（<20%对象）；更改对象可见性或LOD级别 |
| **高频** | 渲染对象 |

**问题**：所有这些都在绘制循环内完成吗？

#### HypeHype解决方案：将低频操作与绘制分离

```
┌─────────────────────────────────────────────────────────────┐
│  PSO (管线状态对象)                                           │
├─────────────────────────────────────────────────────────────┤
│  • 应用启动时构建所有管线（所有渲染状态组合）                  │
│  • 将PSO句柄存储到每个对象的视觉组件                          │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  绑定组 (Bind Groups)                                         │
├─────────────────────────────────────────────────────────────┤
│  • 关卡加载时为每个材质创建绑定组：包含所有纹理和缓冲区绑定     │
│  • 将材质绑定组句柄存储到每个对象的视觉组件                   │
│  • 更改材质 = 单个Vulkan/Metal命令                            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  数据上传                                                     │
├─────────────────────────────────────────────────────────────┤
│  持久数据：                                                   │
│  • 启动时上传一次                                             │
│  • 数据变化时增量更新                                         │
│                                                               │
│  动态数据：                                                   │
│  • 批量上传整个通道：没有每绘制映射和取消映射                  │
│  • 按频率分离：每通道 | 每绘制                                 │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  资源同步                                                     │
├─────────────────────────────────────────────────────────────┤
│  • 渲染通道：RT纹理转换到写入然后读取                          │
│  • 没有每绘制调用的状态跟踪                                    │
└─────────────────────────────────────────────────────────────┘
```

## 四、实现阶段 (Implementation)

### 4.1 快速安全的对象生命周期跟踪

#### 现代实践的问题

| 方法 | 问题 |
|------|------|
| 智能指针、引用计数、RAII | **太慢**：每对象内存分配、数据分散导致缓存未命中、拷贝指针=2x原子操作 |
| | **安全问题**：迭代数组时引用计数归零导致析构器RAII副作用、使用互斥锁杀死性能 |

#### HypeHype解决方案：数组！

```
┌─────────────────────────────────────────────────────────────┐
│  数组方案                                                    │
├─────────────────────────────────────────────────────────────┤
│  • 同一类型所有对象的一次大分配                               │
│  • 数组索引是优雅的数据句柄                                   │
│    - POD，易于拷贝和传递                                      │
│  • 安全地传递给工作线程                                       │
│    - 无法解引用数组索引，需要访问数组                         │
└─────────────────────────────────────────────────────────────┘
```

#### 池和句柄 (Pools and Handles)

```
池结构：
┌─────────┬─────────┬─────────┬─────────┬─────────┬─────────┐
│  data   │  data   │  data   │  data   │  data   │  data   │
├─────────┼─────────┼─────────┼─────────┼─────────┼─────────┤
│  gen 1  │  gen 1  │  gen 1  │  gen 5  │  gen 2  │  gen 1  │  ← 每个槽位的代计数器
└─────────┴─────────┴─────────┴─────────┴─────────┴─────────┘
    0         1         2         3         4         5

空闲列表：未使用池索引的数组（栈）
• 删除对象 = 压入索引
• 创建对象 = 弹出索引，需要时调整大小（无指针→安全）

句柄结构：
┌─────────────────┬─────────────────┐
│   数组索引      │    代计数器     │
│    (32位)       │    (32位)       │
└─────────────────┴─────────────────┘

pool.get<T>(handle): 比较代计数器，不匹配→返回null
```

**弱引用语义**：
- 空检查（可预测分支）在现代CPU上几乎免费
- 比多线程系统中的回调好得多：没有竞争/互斥锁！

#### 热数据 vs 冷数据分离

```
┌─────────────────────────────────────────────────────────────┐
│  问题：易用API需要辅助数据                                    │
├─────────────────────────────────────────────────────────────┤
│  纹理不能只是VkTexture或MTL::Texture                          │
│  • 额外数据：大小、格式、数据指针、分配器...                  │
│  • 低频任务需要：更新、回读、同步、创建依赖资源、释放内存     │
├─────────────────────────────────────────────────────────────┤
│  渲染只需要热数据                                             │
│  • 辅助数据膨胀结构 → 在性能关键的绘制循环中伤害缓存          │
└─────────────────────────────────────────────────────────────┘
```

**HypeHype解决方案：池内热数据和冷数据分离**
- 池有两种类型和两个数组：热和冷
- 两者可以用相同的句柄访问（使用相同的数组索引）
- 分离热数据和冷数据（调查）→ 避免妥协！
- 使用SoA（Structure of Arrays）布局

### 4.2 快速干净的C++20 API用于构建资源

#### Vulkan和DirectX的问题
- 使用大结构初始化复杂资源
- 结构包含其他结构和非拥有指针引用结构数组
- 代码膨胀、没有默认值、临时对象生命周期导致bug

#### 现有解决方案的问题
- **构建器模式**：调试性能糟糕，发布代码生成也不理想

#### HypeHype解决方案：C++20指定结构初始化器
```cpp
// 最好的C99特性终于在C++中出现了！等了11年！
// 默认值由C++11结构聚合初始化提供
// 极其干净的语法，最佳可读性

// 数组数据？
// • 支持初始化列表的自定义span
// • 安全："const &&"参数强制临时对象
```

### 4.3 高效的GPU内存分配

#### 临时分配（高频）
```
┌─────────────────────────────────────────────────────────────┐
│  要求：必须极快（每帧百万次调用）                             │
├─────────────────────────────────────────────────────────────┤
│  • 碰撞分配128MB内存块（存储在环中）                          │
│    - 后端堆对象包含1个完整大小的GPU缓冲区：Buffer = 偏移 + 堆索引│
│  • 后端提供具体的碰撞分配器对象                               │
│    - 分配函数碰撞指针，内联到调用者                           │
│    - 检查偏移 >= 128MB → 调用后端提供下一个块                 │
│    - WebGL2：32MB CPU内存块，每渲染通道一次glBufferSubData调用 │
└─────────────────────────────────────────────────────────────┘
```

#### 持久分配（仅在需要时）
```
┌─────────────────────────────────────────────────────────────┐
│  两级分离适应算法 (Two-Level Segregated Fit)                  │
├─────────────────────────────────────────────────────────────┤
│  • O(1)硬实时分配/释放                                        │
│    - 使用两级位域 + 2x lzcnt查找桶                            │
│  • 删除：合并两侧相邻块（如果它们空闲）                        │
│  • Metal（placement heaps）和Vulkan使用相同的分配器！         │
│  • 已在GitHub开源（MIT许可证）                                │
└─────────────────────────────────────────────────────────────┘

开源地址：https://github.com/sebbbi/OffsetAllocator
```

### 4.4 绑定组暴露给用户层

#### 传统方式的问题
```
┌─────────────────────────────────────────────────────────────┐
│  传统方式：分离绑定                                           │
├─────────────────────────────────────────────────────────────┤
│  • 后端按需创建新绑定组                                       │
│  • 问题：创建新组很昂贵                                       │
│  • 变通方案：哈希映射中存储绑定组 → 慢！                      │
└─────────────────────────────────────────────────────────────┘
```

#### HypeHype解决方案：用户层绑定组

```
HypeHype绑定组层级：

┌─────────────────┐
│  渲染通道全局    │  ← Slot 0
├─────────────────┤
│     材质        │  ← Slot 1
├─────────────────┤
│  着色器特定     │  ← Slot 2
├─────────────────┤
│  动态绘制数据   │  ← Slot 3 (动态偏移)
└─────────────────┘
      ↑
  非硬编码！
```

**核心设计**：
- 用户从一组绑定构建不可变的持久绑定组
  - 示例：材质（5个纹理 + 带值数据的统一缓冲区）
- 绘制调用有三个绑定组槽位：0, 1, 2（Vulkan Android最低规格=4）
  - 匹配GLSL着色器描述符集槽位
  - 按绑定频率分组数据

**动态绑定组抽象**：
- 提供绘制数据的灵活方式，仅支持缓冲区绑定（带偏移）
- Vulkan/WebGPU：set 3（动态偏移）
- Metal：setBuffer + setOffset
- Push constants？在许多移动GPU上模拟 :(

### 4.5 软件命令缓冲区——但快一个数量级

#### 初始设计：绘制结构数组
```
┌─────────────────────────────────────────────────────────────┐
│  初始设计：绘制结构数组                                       │
├─────────────────────────────────────────────────────────────┤
│  • 只包含"元数据"                                             │
│  • 超级简单快速                                               │
│    - 64字节 = 每绘制1个缓存行                                 │
│  • 实际数据在缓冲区内部（组内）                               │
│    - 从N个线程直接写入临时数据到GPU内存                       │
└─────────────────────────────────────────────────────────────┘
```

#### 数据分析
```
• 所有字段都是32位整数
• 分箱内容渲染时，大多数数据在绘制调用之间不变
• 平均4.5个字段变化（~18字节）
```

#### 绘制流 (Draw Stream)：绘制调用的数据接口

```
绘制结构布局（64字节）：
┌────────┬────────┬────────┬────────┬────────┬────────┐
│shader  │bind    │index   │vertex  │tri     │dynbuf  │
│16+16   │group   │buffer  │buffer  │count   │offset  │
│        │16+16   │16+16   │16+16   │32      │32      │
└────────┴────────┴────────┴────────┴────────┴────────┘

优化：只存储绘制结构的修改字段
• uint32脏掩码告诉哪些字段已修改
```

#### 用户层：绘制流写入器类
- 包含一个绘制结构（当前状态）+ 脏掩码
- 绘制：在前面写入脏掩码（存储偏移）
- 每个字段的设置器：如果改变 → 设置脏位 + 写入字段到流

#### 后端：流解码
```
对每个绘制：
  1. 读取脏字段位掩码
  2. 对每个设置的位：读取字段并发出图形API调用

优势：
  • 后端没有变化跟踪
  • ~3倍减少带宽
```

## 六、关键要点总结

### 6.1 研究阶段要点
1. **数据驱动决策**：收集详细的硬件分析数据，基于95%用户覆盖率设定最低规格
2. **硬件现实**：移动GPU与桌面/主机GPU架构差异巨大，不能直接移植现代PC渲染技术
3. **厂商合作**：与硬件厂商建立联系，获取最佳实践指导

### 6.2 设计阶段要点
1. **最小抽象**：薄的底层API包装器，找到功能共同集合，最优抽象差异
2. **迭代设计**：先写模拟用户代码和模拟API，迭代满意后再实现后端
3. **频率分离**：将低频操作（PSO创建、绑定组设置）与高频绘制分离

### 6.3 实现阶段要点
1. **数组+句柄**：避免智能指针开销，使用数组索引+代计数器实现安全引用
2. **热冷分离**：将频繁访问的热数据与辅助冷数据分离，优化缓存
3. **C++20特性**：利用指定初始化器实现干净高效的资源构建API
4. **高效分配**：临时内存使用碰撞分配，持久内存使用两级分离适应算法
5. **用户层绑定组**：将绑定组创建暴露给用户，避免运行时哈希查找
6. **绘制流优化**：只记录变化的绘制字段，减少带宽使用
